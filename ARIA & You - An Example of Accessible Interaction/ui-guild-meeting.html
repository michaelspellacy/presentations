ARIA & You: An Example of Accessible Interaction

Posted on March 21st, 2017 by Michael Spellacy

ARIA (Accessible Rich Internet Applications) is an extension of HTML, created by the W3C, that allows authors to make their websites or appications more usable to various assistive technologies (AT), such as screen or braille readers.

Today we will review the navigation for Kaiser Permanent's career site and discuss the some of the finer points of including ARIA in our work.

In the process of creating our navigation, we adhere to the core principles of progressive enhancment and resilient design, and that is in crafting an experience that is always usable at it's core, while enahncing it for more capable technology. As always, we begin with good old HTML.

Semantics

For the markup of our navigation, we focus on providing semantic robustness:

  1. Use of HTML5 landmarks, such as <navigation>, provide additional baked in information to AT users, so be sure to use them where appropriate.
  2. While sighted users are instantly provided with context when looking at the screen, AT users may require additional aids to assist them in using your interface. For example, we could create a heading and then hide it offscreen, but ARIA can provide the same information about your interface, through use of the aria-label attribute. On our navigation we have added such a label and have given it a value of "primary".
  3. ID's on our <li> elements will provide a nice hook in helping us additional context and enhancment for sighted and non-sighted users alike.
<nav id="primary-nav" aria-label="primary">

    <ul id="primary-nav-list">
    <li>

        <h2 class="primary-nav-heading">Who We Are</h2>

        <ul>
            <li id="nav-the-kp-difference"><a href="/the-kp-difference/">The KP Difference</a></li> 
            <li id="nav-location"><a href="/locations/">Our Locations</a></li>
            <li id="nav-mission-and-history"><a href="/mission-and-history/">Mission & History</a></li>
            <li id="nav-innovation"><a href="/innovation/">Innovation</a></li>
        </ul>

    </li>

    ...

    </ul>

</nav>

Enhance

Since we will be using JavaScript to enhance our navigation, we need to ensure that we are providing parity for AT users. It is not enough to create an interface that can be toggled visually. There is deeper work to be done here to provide a navigational system that can be used by all people.

  • First, we add our menu button. This button will toggle the navigation opened and closed. We could add the button directly to our markup as inline HTML, but since it only exists to toggle our navigation, via script, then it can be treated as an enhancement, via script, as well.

    Note the addition of aria-expanded to the button. Since our navigation is initially closed, we immediatly set this value to a default state of "false".

    <button id="primary-button" aria-expanded="false">Menu</button>

    Normally, we might place a menu button outside of the navigation landmark, but by placing it within the landmark, we are providing better context for AT users. Screen readers will now announce a varient of: "Menu, collapsed button navigation primary". This allows our button to dovetail nicely with the parent aria-label attribute that has a value of "primary". If this button were placed outside of our navigation, a screen reader might only announce: "Menu, collapsed button", which is not very helpful. In that case, we might require additional ARIA attributes, such as aria-labeledby, to auditorially bind the button and the navigation together, but why do that when simply inserting the button here is so much easier and elegant?

  • Next, we need to enhance our h2 element. It is no longer a heading, but an interactive element within our navigation.:

    <h2 class="primary-nav-heading" aria-expanded="false" role="button" tabindex="0">Who We Are</h2>

    Here we must also include aria-expanded with an initial state of "false". We also append a role attribute with a value of "button", since it is now a button, and a tabindex with a value of "0". Setting an element with a tabindex of zero allows a non-focusable element to receive focus.

  • Finally, we append a hidden element to the navigation:

    <div hidden id="current-selection">Current Page</div>

    This element will work cosely with our ID's.

Interaction

Business as usual in applying a script that toggles our navigation button and navigation sub menus, but in additon to these behaviors, we also need to ensure that we are applying equivelency to our ARIA attributes. ARIA states need to be toggled as well, to indicate to a screen reader that states within the menu have changed.

A part of our script will also match the <body> ID with the <li> ID. When there is a match, our script will append aria-labeledby with a value of "current-selection", to the list items hyperlink:

<a href="#content" aria-describedby="current-selection">The KP Difference</a>

In addition to this, we will also change the matched links href value to instead anchor to a position within the page, rather than take us to the same page again after clicking on it. When a screen reader accesses the link, the following will be read aloud: "Internal, link, The KP Difference" or "Link, The KP Difference, Current Page". The aria-describedby value can also be used as a styling hook to indicate to visual users that the selection is for the current page as well. This enhancement is a really nice touch for visual and non-visual users alike!

Don't Forget

  • Escape Key - Be mindul of keyboard navigation. This means providing additional scripting for events such as clicking on the escape key or space bar.

  • Focus Management - What happens when a user closes your interface? Where does the elements focus go? Typically we return focus to the initiation point, so that a user can continue on with experience from there. So, closing the navigation with an escape key, might bring us back to the menu button, etc.

Takeaways

  • Our interfaces go deeper than what we can see on our monitors. A great user interface can be used by anynody.
  • Parity between the visual and non-visual experience is integral to success.
  • Use the right elements for the job and use ARIA to aid the experience where needed. The first rule of ARIA is not to use it. For example, use <button> instead of <div role="button">
  • Accessibility can be hard at first, but get's easier with practice. Take simple steps to improving. Estimate accordingly to allow for it.
